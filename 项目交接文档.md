## 1. 项目目标

本项目目标是做一套滑轨式水下机器人控制系统，用于让机器人沿滑轨跟随游泳运动员运动。系统由电脑 APP、Arduino 控制器、MCP2515 CAN 模块、48V 一体化伺服电机和 48V 动力电池组成。

整体分工如下：

| 层级 | 负责内容 |
| --- | --- |
| 电脑端 APP | 摄像头画面、运动员跟踪、速度估算、定速巡航、参数输入、安全状态显示 |
| Arduino Mega 2560 | 串口接收上位机命令、本地 PID 转速闭环、CAN 报文发送和反馈读取 |
| MCP2515 CAN 模块 | Arduino SPI 与电机 CAN 总线之间的转换 |
| 48V 伺服电机 | 接收 CAN 速度命令并驱动滑轨机构 |

当前设计原则是：APP 生成目标速度，Arduino 做本地 PID，电机执行速度命令。不要在 APP 和 Arduino 两边同时做两套 PID。

## 4. 硬件接线交接

### 4.1 MCP2515 与 Arduino Mega 2560

当前代码以 Mega 2560 的硬件 SPI 为准，不能按 Uno 的 D11/D12/D13 接。

| MCP2515 引脚 | Mega 2560 引脚 | 说明 |
| --- | --- | --- |
| VCC | 5V | MCP2515 模块供电 |
| GND | GND | 控制地 |
| CS | D10 | 代码中 `MCP2515_CS_PIN = 10` |
| SO / MISO | D50 | Mega 硬件 SPI |
| SI / MOSI | D51 | Mega 硬件 SPI |
| SCK | D52 | Mega 硬件 SPI |
| INT | D2 | 当前代码保留接线，但不依赖中断 |

INT 接 D2 是可以的，但当前 INO 使用轮询方式读取 CAN，不靠 D2 中断。真正的数据链路是 D50/D51/D52 的 SPI 到 MCP2515，然后由 MCP2515 的 CANH/CANL 到电机。

### 4.2 电机、电源与 CAN

- 电机动力端接 48V 电池正负极。
- 48V 电源绝对不能接 Arduino 的 5V、VIN 或任何 IO。
- 电机 CANH 接 MCP2515 CANH，电机 CANL 接 MCP2515 CANL。
- 48V 电池负极、电机控制参考地、Arduino/MCP2515 GND 需要可靠共地，建议单点共地。
- CANH/CANL 建议用双绞线，尽量远离电机动力线。
- CAN 总线两端各有一个 120Ω 终端电阻；断电测 CANH-CANL，双端终端正常约 60Ω。
- 电机拨码应为 ID1，也就是 `001`。

## 7. APP 运行模式

### 7.1 定速巡航

定速巡航用于输入一个机器人目标线速度，让机器人保持该速度，直到点击停止或安全保护触发。

换算关系：

```text
目标RPM = 电机方向符号 × 目标线速度 × RPM/(米/秒)
```

特点：

- 不依赖摄像头。
- 仍依赖电机反馈。
- APP 固定发送目标 RPM。
- Arduino 根据实际 RPM 做本地 PID。
- 运行中可以通过“停止电机”退出。

### 7.2 视觉跟随

视觉跟随用于根据摄像头画面估算运动员速度，让机器人跟随运动员。

计算关系：

```text
相对位移 = 画面方向符号 × 像素偏移 / 像素每米
机器人速度 = 电机方向符号 × 实际RPM / RPM每米每秒
运动员速度 = 机器人速度 + 相对速度
目标RPM = 电机方向符号 × 运动员速度 × RPM每米每秒
```

操作顺序：

1. 连接电机后端。
2. 确认标定参数。
3. 打开摄像头。
4. 手动框选运动员。
5. 目标锁定后启动视觉闭环。
6. 异常时点击停止或使用物理急停。

## 8. 标定参数说明

| 参数 | 含义 | 如何获得 |
| --- | --- | --- |
| 像素/米 | 画面中 1 米对应多少像素 | 在滑轨方向放置已知长度标尺实测 |
| RPM/(米/秒) | 机器人 1 m/s 对应多少电机 RPM | 低速运行，记录位移和时间后换算 |
| RPM 绝对上限 | APP 允许下发的最大绝对 RPM | 不超过协议上限，实机应保守 |
| RPM 变化率/秒 | 目标转速每秒最大变化 | 用于软启动，避免突变 |
| 画面向右代表 | 画面 x 正方向对应实际轨道方向 | 现场实测选择 `+1` 或 `-1` |
| 正 RPM 代表 | 电机正转对应实际轨道方向 | 低速点动确认 |

默认参数只是占位，不能直接当作真实标定值使用。

## 9. 安全逻辑交接

APP 状态：

| 状态 | 含义 |
| --- | --- |
| `DISCONNECTED` | 未连接 |
| `STOPPED` | 后端已连接，电机停止 |
| `CAMERA_READY` | 摄像头已打开 |
| `TARGET_LOCKED` | 目标已框选并锁定 |
| `RUNNING` | 正在定速或视觉闭环运行 |
| `FAULT` | 故障锁定，必须人工确认 |

主要安全策略：

- APP 启动真实后端前会有风险确认。
- APP 连接后先发送停止或零速。
- APP 有独立监督进程，GUI 卡死时仍尝试停车。
- Arduino 500ms 没收到新控制命令会停车。
- Arduino 200ms 没收到 CAN 反馈会停车。
- CAN 连续发送失败会锁定停止。
- 视觉目标丢失、位移超出安全区域、摄像头异常会触发故障。
- 进入 `FAULT` 后不会自动恢复，必须人工确认。

重要：软件安全不能替代物理急停、机械限位和驱动器保护。

## 10. 实机排查顺序

如果电机不转，优先按下面顺序查，不要一开始就调 PID。

1. 串口是否打印 `CAN总线初始化完成（1Mbps，16MHz晶振，标准帧）`。
2. MCP2515 晶振是否确认为 16MHz。
3. 代码是否使用 `MCP_16MHZ`。
4. Mega SPI 是否接 D50/D51/D52，CS 是否接 D10。
5. CANH/CANL 是否接反。
6. 断电测 CANH-CANL 是否约 60Ω。
7. 电机拨码是否 ID1。
8. 是否持续打印“等待ID1的0x208转速反馈”。
9. 有 USB-CAN 或示波器时，抓线看是否有 `0x300`、周期 `0x202` 和返回 `0x208`。

判断标准：

- 如果串口一直显示目标有数、实际为 0、输出变大，只能说明 PID 在计算，不能证明 CAN 已打通。
- 第一目标是稳定收到 `0x208`，不是直接让电机高速转起来。
- 没有 `0x208` 时，当前安全逻辑会拒绝启动或超时停车。

## 12. 后续开发建议

短期建议：

- 把 INO 初始 `targetRPM` 从 300 改为 0，降低人工串口误启动风险。
- 明确禁止 `-1 RPM`，因为协议中 `0xFFFF` 表示无效，而 `-1` 的 16 位补码也是 `0xFFFF`。
- 增加 `0x209` 解析，显示电压、温度、故障位。
- 记录 CAN 报文日志，方便后续排故。

中期建议：

- APP 与 Arduino 建立更明确的版本号和协议版本。
- APP 增加实机验收记录导出。
- 用配置文件保存不同场地的标定参数。
- 加入轨道限位输入，形成软硬件双限位。

长期建议：

- 从手动 CSRT 框选升级到自动目标检测。
- 优化水下反光、水花和遮挡下的跟踪稳定性。
- 增加多传感器冗余，例如编码器、限位开关或 IMU。
- 做完整实验数据曲线：目标 RPM、实际 RPM、机器人速度、运动员估算速度、误差。
