# 游泳滑轨机器人视觉闭环设计

日期：2026-07-15

## 1. 目标与边界

本设计以 `motor_control/pidnew2_copy_20260714231054.ino` 的串口和 CAN 行为为控制依据，重新构建 Python 视觉应用，并将视觉测速接入电机控制形成单轴闭环。

系统只控制滑轨前后一个方向。用户在画面中手动框选游泳运动员，程序跟踪目标中心相对画面中央十字准星的横向偏移，并结合电机反馈速度估算运动员相对地面的速度。Python 不实现 PID；现有 INO PID 和电机驱动器内部 PID 保持原有职责。

摄像头型号和接口暂不固定。APP 提供手动摄像头源输入，支持摄像头索引、视频设备 URL 或 OpenCV 可接受的设备字符串。录制视频可用于离线视觉验证，但不得启用真实电机闭环。

## 2. 已确定的运动学

摄像头固定在滑轨机器人上。沿滑轨统一定义一个带符号的正方向：

```text
v_relative = v_swimmer - v_robot
v_swimmer = v_robot + v_relative
```

视觉测得目标相对准星的带符号横向偏移：

```text
offset_px = target_center_x - frame_center_x
offset_m = camera_axis_sign * offset_px / pixels_per_meter
```

通过最近一段 `(monotonic_time, offset_m)` 样本的线性拟合斜率计算相对速度，避免直接相邻帧差分放大跟踪框抖动：

```text
v_relative = slope(offset_m versus monotonic_time)
v_robot = motor_axis_sign * actual_rpm / rpm_per_mps
v_swimmer = v_robot + v_relative
target_rpm = round(motor_axis_sign * v_swimmer * rpm_per_mps)
```

`camera_axis_sign` 和 `motor_axis_sign` 只能取 `+1` 或 `-1`。`pixels_per_meter` 和 `rpm_per_mps` 必须为有限正数。

Python 不对中心偏移执行 P、I 或 D 修正。十字准星用于建立偏移坐标、计算相对速度、显示位置和执行越界保护。闭环目标是使机器人与运动员共速，而不是额外建立第三层 PID。

## 3. 标定与默认值

- `pixels_per_meter` 默认值为 `120.0`。
- `rpm_per_mps` 默认占位值为 `1.0`。
- 两个方向参数默认值为 `+1`。
- RPM 协议绝对上限为 `2047`。

默认换算值仅用于让界面和离线计算具备完整数据结构，不代表真实机械标定。用户必须主动勾选“换算值已经实测确认”，APP 才能允许真实电机闭环启动。修改任一换算或方向参数后，该确认自动失效，必须重新确认。

## 4. 软件模块

```text
vision_app/
├── swimming_gui.py       唯一启动入口
├── swimming_app.py       Tkinter 界面、提示和状态协调
├── vision_tracker.py     摄像头、手动 ROI、CSRT 和单轴测量
├── control_core.py       纯速度解算、换算、限幅和变化率限制
├── motor_link.py         pyserial 连接、命令编码和遥测解析
├── settings.py           类型化配置、校验及安全确认状态
└── tests/
    ├── test_control_core.py
    ├── test_motor_protocol.py
    └── test_safety_state.py
```

### 4.1 `vision_tracker.py`

- 将纯数字摄像头源解释为设备索引，其余内容作为设备字符串或 URL 交给 OpenCV。
- 打开源后读取首帧，由用户手动框选唯一目标。
- 使用 CSRT 更新目标框，但控制计算只使用目标框中心横坐标。
- 始终绘制画面中央十字准星、目标中心、偏移方向和安全区域。
- 选框后的第一帧只建立时间和位置基准，不生成相对速度。
- 跟踪失败、帧中断、时间戳无效或目标越界时返回明确的无效测量，不沿用最后一次测量。

### 4.2 `control_core.py`

- 不依赖 Tkinter、OpenCV 或串口，确保全部计算可独立测试。
- 维护有限长度和有限时间跨度的位置样本。
- 至少获得三个有效样本后才计算线性拟合速度。
- 拒绝 `NaN`、无穷值、非递增时间戳和非法标定值。
- 根据电机反馈计算机器人速度，再按已确定公式计算运动员速度。
- 将运动员速度换算为目标 RPM，执行协议限幅和每周期变化率限制。
- 输出结构同时包含原始值、限制后的值、饱和标志和不可输出原因。

### 4.3 `motor_link.py`

- 串口固定使用 `115200` 波特率，与 INO 一致。
- 编码以下带换行命令：`S`、`P`、`T<signed_integer>`。
- 只允许 `T` 参数位于 `-2047..2047`，范围外值不得编码。
- 解析 INO 的 `目标:<value>,实际:<value>,输出:<value>` 遥测行。
- 正确处理串口分片、空行、启动提示、未知行和无效数值；未知行可记录但不得更新有效反馈时间。
- 串口读取在线程中执行，通过线程安全队列将事件传给 Tkinter 主线程；工作线程不得直接操作控件。
- 关闭串口前尽力发送 `P`。

### 4.4 `swimming_app.py`

- 只负责编排组件、界面刷新和安全状态机，不重复实现速度公式或协议解析。
- 控制命令以 20 Hz 更新，使用最新且仍有效的视觉测量与电机反馈。
- UI 刷新和控制发送使用独立计划任务，均运行在 Tkinter 主线程。
- 任何未处理异常都先触发最佳努力停机，再显示故障并安全关闭资源。

## 5. APP 提示要求

APP 必须主动告诉用户当前状态、下一步操作和不能启动的具体原因。

### 5.1 常驻提示

窗口顶部设置高可见状态横幅：

- 灰色：尚未连接，并提示“请先填写串口和摄像头源”。
- 蓝色：已连接但未选目标，并提示“请打开摄像头并框选运动员”。
- 黄色：条件未满足，逐项列出缺失条件，例如反馈未收到、标定未确认或方向未验证。
- 绿色：闭环运行，显示当前目标 RPM 和反馈新鲜度。
- 红色：故障锁定，持续显示首要故障原因和“电机已请求停止”。

按钮禁用时，状态区必须解释禁用原因，不能让用户通过反复点击猜测操作顺序。

### 5.2 启动确认

首次启动真实闭环前弹出确认窗口，汇总：

- 串口与摄像头源；
- `pixels_per_meter` 和 `rpm_per_mps`；
- 两个方向配置；
- 当前实际 RPM、相对速度和即将发送的目标 RPM；
- 强电机风险提示；
- “急停/停止”操作位置。

只有用户明确确认后才依次发送首个 `T` 和 `S`。取消确认保持停止状态。

### 5.3 故障提示

故障发生时同时提供：

- 红色常驻故障横幅；
- 一次性弹窗，说明发生了什么、是否成功发送 `P`、如何恢复；
- 可复制的详细故障文本和时间；
- “确认故障”按钮。

关闭弹窗不能恢复运行。故障解除后仍必须重新框选目标并手动启动。

正常状态变化使用非阻塞状态提示，避免频繁弹窗影响操作。只有真实启动确认和故障锁定使用模态弹窗。

## 6. 安全状态机

状态定义：

```text
DISCONNECTED
STOPPED
CAMERA_READY
TARGET_LOCKED
RUNNING
FAULT
```

允许的主路径：

```text
DISCONNECTED -> STOPPED -> CAMERA_READY -> TARGET_LOCKED -> RUNNING
```

约束：

- 建立串口连接后立即发送 `P`，随后进入 `STOPPED`。
- 只有串口有效、遥测新鲜、摄像头有效、目标有效、标定已确认、方向有效且控制结果有限时才可进入 `RUNNING`。
- 人工停止在所有状态下具有最高优先级。
- `RUNNING` 中摄像头断流、跟踪丢失、视觉测量过期、电机反馈过期、串口异常、参数变化、非法计算或目标越出安全区域，立即尽力发送 `P` 并进入 `FAULT`。
- `FAULT` 不允许自动恢复；确认故障后返回 `STOPPED`，目标锁定和速度样本全部清空。
- 断线重连只返回 `STOPPED`，绝不恢复旧目标速度。

视觉和遥测新鲜度阈值作为经过校验的配置常量；初始实现分别采用 250 ms 和 350 ms。目标命令每 50 ms 更新一次。

## 7. Arduino 失联看门狗

保留现有 CAN ID、帧布局、PID 参数和 10 ms 发送周期，仅增加独立通信安全逻辑：

- 记录最后一次有效 `S` 或合法 `T` 命令的 `millis()`。
- `motorEnable == true` 且超过 500 ms 未收到上述命令时：
  - 立即将 `motorEnable` 设为 `false`；
  - 将目标和 PID 输出置为安全零值并重置 PID 状态；
  - 向电机持续发送零速；
  - 串口打印可解析的看门狗停机提示。
- 通信恢复不能自动重新使能，必须再次收到 `S`。

Python 在 `RUNNING` 状态每 50 ms 发送新的 `T` 命令，因此正常通信下不会触发 500 ms 看门狗。USB 断开或 Python 崩溃时，MCU 独立完成停机。

## 9. 测试设计

### 9.1 控制核心单元测试

- 静止、同向、反向和机器人/运动员共速。
- `v_swimmer = v_robot + v_relative` 的全部符号组合。
- RPM 与线速度换算的方向和往返一致性。
- 首帧无输出、样本不足、非递增时间和窗口淘汰。
- 线性轨迹、带小幅抖动轨迹和异常跳点。
- `NaN`、无穷值、零或负标定值。
- RPM 上下限和目标变化率限制。

必须包含以下基准断言：

```text
v_robot=0.0,  v_relative=+0.5 -> v_swimmer=+0.5
v_robot=+1.0, v_relative=-0.2 -> v_swimmer=+0.8
v_robot=-0.7, v_relative=+0.1 -> v_swimmer=-0.6
v_relative=0.0                 -> v_swimmer=v_robot
```

### 9.2 协议测试

- `S\n`、`P\n` 和正负 `T` 命令精确编码。
- 拒绝小数 RPM、范围外 RPM 和非有限输入。
- 完整、分片、连续和带噪声的中文遥测解析。
- 未知启动消息不更新电机反馈新鲜度。

### 9.3 状态机与集成测试

- 覆盖所有允许状态转换和禁止转换。
- 使用假串口验证连接先发 `P`、启动顺序为 `T` 后 `S`、运行期间 20 Hz 更新。
- 验证摄像头丢失、跟踪丢失、反馈超时、串口断开、配置变化和程序关闭均触发停机。
- 模拟 Python 心跳停止，验证 Arduino 看门狗逻辑在 500 ms 后进入停机状态。
- 通过可注入的合成视觉测量完成无电机闭环测试。

## 10. 真实硬件验收

软件测试通过不等于真实电机安全验收完成。硬件验收顺序固定为：

1. 电机动力断开或滑轨架空，验证串口命令、状态提示和故障停机。
2. 使用最低可辨识 RPM 验证电机方向和摄像头方向。
3. 主动拔除 USB，验证 Arduino 在 500 ms 内停机。
4. 遮挡目标和断开摄像头，验证 APP 进入故障且不自动恢复。
5. 实测 `rpm_per_mps` 和 `pixels_per_meter`，填写真实值并重新确认标定。
6. 逐级提高速度，检查饱和、变化率限制和机械边界。

完成上述步骤前，不得声明真实闭环已经通过安全验证。

## 11. 完成标准

- Python 控制逻辑完全依据已确认 INO 串口协议。
- 旧 Python PID 和重复视觉脚本不再存在。
- APP 能手动填写摄像头源、手动框选目标并显示中心准星。
- APP 能读取实际 RPM、计算带符号相对速度和运动员速度，并发送限幅目标 RPM。
- Python 中不存在 PID 控制器。
- 全部故障路径都会请求停止，INO 具备独立失联看门狗。
- APP 对操作顺序、启动风险、禁用原因和故障恢复提供明确中文提示。
- 自动测试全部通过，真实硬件验收步骤有明确记录且不会被软件测试替代。
