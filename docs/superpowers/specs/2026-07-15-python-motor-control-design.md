# Python 电机控制与安全监督设计

日期：2026-07-15  
状态：待用户书面复核

## 1. 目标与范围

本阶段把已经验证可执行的 Arduino INO 控制逻辑移植为 Python，实现视觉应用仅依赖软件即可完成“目标速度计算—电机命令输出—速度反馈—安全停车”的闭环链路，同时保留现有 Arduino 串口方案作为兼容后端。

当前电脑没有可识别的 Arduino、USB-CAN 或其他 CAN 适配器，因此本阶段可以完成协议、控制器、监督进程、虚拟 CAN 仿真和界面集成验证，但不能宣称真实 USB-CAN 设备或真实电机已经通过联调。真实设备验证结果必须在界面和交付说明中明确标记为“未验证”。

不在本阶段实现视觉外环 PID。运动员速度仍按现有视觉链路计算：相机测得运动员相对机器人的位移和速度，并使用

`运动员绝对速度 = 机器人当前速度 + 运动员相对机器人速度`

解算机器人目标速度。换算系数使用默认值，并继续由界面向用户提供可编辑输入。电机内部已有 PID，因此默认直接向驱动器发送目标转速。

## 2. INO 协议的 Python 等价实现

Python 实现保持 `pidnew2_copy_20260714231054.ino` 的 CAN 定义：

- CAN 波特率：1,000,000 bit/s
- 电机编号：2
- 转速命令帧 ID：`0x202`
- 激活帧 ID：`0x300`
- 反馈帧 ID：`0x208`
- 驱动输出范围：`-2047..2047`
- 主控制周期：10 ms

帧编码规则：

- 激活帧 DLC 为 8，初始 8 字节均为 `0xFF`，再将 `data[motor_id - 1]` 写为 `0x00`。
- 转速帧 DLC 为 8，初始 8 字节均为 `0xFF`。目标值转换为有符号 16 位整数，按大端序写入偏移 `(motor_id - 1) * 2` 的两个字节。
- 反馈只接受 ID `0x208` 且 DLC 不小于 6 的帧；`data[4]`、`data[5]` 按大端有符号 16 位整数解析为实际转速。
- 超范围输出先限制到 `-2047..2047`；NaN、无穷值、错误长度、错误 ID 和过期数据均不得下发。

控制器启动后先建立后端，再发送激活帧和至少 3 个零速帧。只有收到新鲜、有效的反馈并完成显式解锁后才允许发送非零命令。停止时发送至少 5 个零速帧，每帧间隔 10 ms。

Python API 同时保留 INO 串口命令的语义映射：

- `S`：启动/解锁
- `P`：停止并清零
- `T`：设置目标转速
- `KP`、`KI`、`KD`：设置兼容 PID 参数

直接 CAN 后端不需要把这些命令编码为文本，但上层调用和串口兼容后端应具有一致的状态含义。

## 3. 控制模式

系统提供两种明确模式：

### 3.1 驱动器内部 PID 模式（默认）

上层计算出的目标 RPM 经限幅后直接写入 `0x202` 帧。Python 不再叠加 PID，以免和电机驱动器内部 PID 形成双重调节。这是日常运行和视觉共速控制的默认模式。

### 3.2 INO PID 兼容模式（默认关闭）

此模式用于复现原 INO 的外部调节结构：

- 设定值：目标 RPM
- 输入值：`0x208` 反馈 RPM
- 输出值：发送给驱动器的 `-2047..2047` 命令
- 默认参数：`Kp=1.0`、`Ki=0.05`、`Kd=0.02`
- 采样周期：10 ms
- 积分采用反计算抗饱和，停止、故障和切换模式时清空积分与微分历史

兼容模式通过固定输入序列的黄金测试验证输出方向、周期、限幅、积分和停止复位行为。由于 Python 实现不直接运行 Arduino 的 `PID_v1_bc` 二进制代码，文档和界面不得将其描述为逐浮点位完全一致；它保证控制结构和可观察行为兼容。

## 4. 软件分层

实现按职责拆分：

- `vision_app/can_protocol.py`：CAN 常量、激活帧/转速帧编码、反馈解码及边界校验。
- `vision_app/motor_backend.py`：统一的 `MotorBackend` 接口，包括连接、激活、设置速度、读取反馈、停止、关闭和健康状态。
- `vision_app/virtual_motor_backend.py`：默认后端，使用 python-can 虚拟总线和独立电机模拟节点完成端到端测试。
- `vision_app/python_can_backend.py`：真实 USB-CAN 后端，接收 python-can 的 `interface`、`channel` 和 `bitrate` 配置。
- `vision_app/motor_link.py`：保留现有 Arduino 串口后端，并适配统一接口。
- `vision_app/pid_compat.py`：可选 INO PID 兼容算法。
- `vision_app/python_motor_controller.py`：10 ms 控制调度、模式切换、反馈新鲜度、目标限幅和状态机。
- `vision_app/motor_supervisor.py`：独立监督进程，唯一持有真实电机后端。
- `vision_app/supervisor_client.py`：GUI 与监督进程之间的本地 IPC 客户端。
- `vision_app/simulated_motor.py`：虚拟电机反馈和故障注入。

统一后端接口位于原始 CAN 帧之上。原因是 Arduino 串口固件只接受 `S/P/T` 等文本命令，并不向 Python 暴露原始 CAN 帧；直接 CAN 后端则复用 `can_protocol.py` 生成和解析帧。两者对上层仍表现为同一套电机操作。

## 5. 后端选择与兼容策略

支持三类后端：

1. `virtual`：默认值，不访问真实硬件，界面持续显示“仿真模式”。
2. `arduino_serial`：复用当前已验证 INO 和串口协议。
3. `python_can`：面向未来 USB-CAN 适配器，配置 python-can 支持的接口名、通道和 1 Mbps 波特率。

程序永远不得因为检测到设备而从虚拟模式自动切换到真实模式。选择任何真实后端都必须由用户主动完成，并再次确认急停/驱动器超时保护已经可用。

python-can 依赖版本约束为 `python-can>=4.4,<5`。适配器型号确定后，只配置该型号在 python-can 中对应的接口和厂商驱动；在没有实物前，不承诺某个厂商驱动已通过验证。

## 6. 独立安全监督进程

真实后端只能由 `motor_supervisor.py` 子进程打开。GUI 不直接持有串口或 CAN 句柄，从而在 GUI 卡死或异常退出时仍能执行停车逻辑。

GUI 每 100 ms 向监督进程发送心跳和最新目标。监督进程使用单调时钟、10 ms 控制循环和本地进程管道，处理以下消息：

- `START`：携带后端和控制配置，启动但保持锁定
- `ARM`：完成安全确认后允许非零输出
- `TARGET`：携带递增序号、单调时间戳和目标 RPM
- `HEARTBEAT`：携带递增序号和单调时间戳
- `STOP`：立即进入零速状态
- `SHUTDOWN`：零速确认后关闭后端和子进程
- `STATUS`：监督进程返回状态、反馈、心跳年龄和故障信息
- `FAULT`：监督进程主动报告不可恢复故障

监督进程拒绝乱序、重复、过期、非有限数值或超范围目标。超过 500 ms 没有收到有效心跳，或目标数据超过 500 ms 未更新时，立即进入故障状态并连续发送零速。发生后端异常、反馈超时、GUI 进程消失或 IPC 关闭时执行相同停车逻辑。

独立进程可以覆盖 GUI 崩溃，但不能在 Windows 整体崩溃、USB 总线掉电或适配器驱动完全失效时保证继续发送 CAN 零速。因此真实运行的硬性前提是电机驱动器自身配置通信超时停车，或现场具备可触达的物理急停。界面必须明确提示这一限制。

## 7. 状态机与停车确认

控制状态为：

`DISCONNECTED -> CONNECTED_SAFE -> ARMED -> RUNNING -> STOPPING -> CONNECTED_SAFE`

任意状态发生严重错误进入 `FAULT`。`FAULT` 状态只允许零速、关闭和显式复位，不允许自动恢复非零输出。

进入真实后端的 `ARMED` 前必须满足：

- 后端连接成功；
- 已发送激活和启动零速帧；
- 收到时间戳新鲜、范围有效的速度反馈；
- 用户已确认急停或驱动器通信超时；
- GUI 和监督进程心跳正常。

正常停车时发送 5 次零速，并等待连续 3 次反馈进入可配置的零速容差。若在停车确认超时内没有反馈或仍在运动，则保持故障、继续尝试零速，不进行应用清理或系统关机。虚拟后端用模拟反馈执行同样的状态流。

## 8. GUI 调整

视觉应用保留手动框选运动员、相对位移测速、换算参数输入和共速计算。新增电机区域显示：

- 当前后端和“仿真/真实”醒目标识
- python-can 接口、通道和波特率输入
- 控制模式：驱动器内部 PID / INO PID 兼容
- 监督进程状态、心跳年龄、目标 RPM、反馈 RPM、最后故障
- 连接、解锁、停车和故障复位按钮
- 未连接串口/CAN、未验证真实设备、没有反馈、心跳超时和物理安全条件的中文提示

真实后端配置不完整、设备不存在或驱动不可用时，连接按钮应失败并给出可执行提示，不能回退后偷偷运行虚拟后端。仿真模式启动时也要显式提示，避免误认为机器人已被控制。

## 9. 配置与持久化

配置保存以下字段：

- 后端类型，默认 `virtual`
- 串口和波特率
- python-can `interface`、`channel`、`bitrate=1000000`
- 电机编号和 CAN ID，默认采用本设计常量
- 控制模式，默认驱动器内部 PID
- PID 兼容参数
- 心跳、反馈和停车确认超时
- RPM 换算系数和用户标定值

真实后端的解锁状态、物理安全确认和上次目标速度不得跨程序启动持久化。每次启动都从零速、未解锁状态开始。

## 10. 验证方案

软件完成前必须通过：

- CAN 编解码黄金测试：激活帧、ID、DLC、字节序、正负边界和错误输入。
- 直接模式和 PID 兼容模式单元测试：周期、限幅、停止复位、反馈过期。
- python-can 虚拟总线端到端测试：控制器、模拟电机、命令帧和反馈帧完整链路。
- 监督进程测试：正常心跳、心跳超时、目标过期、GUI 退出、后端异常和重复零速。
- Arduino 串口兼容回归：现有 `S/P/T` 协议和全部已有测试保持通过。
- GUI 冒烟测试：后端选择、仿真提示、真实模式确认、故障提示和中文显示。
- 依赖安装、静态检查、Python 编译检查和全部测试套件。
- 原 INO 继续保持 Mega2560 编译通过。

由于当前没有真实适配器，验收报告必须把“USB-CAN 真实收发”和“真实电机停车反馈”列为未验证项，而不是跳过后视为成功。
